### Работа с Git: правила ветвления

У нас простая схема ветвления — всего четыре типа веток:

* `main` - стабильная версия продукта. Сюда попадает код после релиза
* `development` - основная ветка разработки. Сюда вливаются все новые фичи и фиксы
* `feature/<название-фичи>` — ветка для разработки новой функциональности (например, `feature/user-profile`, `feature/authorization`)
* `fix/<название-бага>` — ветка для исправления ошибок (например, `fix/login-bug`, `fix/api-timeout`)

### Пошаговая инструкция: начало работы над новой фичей или фиксом

Следуйте этому алгоритму, чтобы правильно начать работу над задачей и избежать конфликтов в коде

#### Шаг 1. Выберите задачу

1. Проверьте, может тимлид уже назначил вам задачи (тогда выполняйте их)
1. Перейдите в раздел **Issues** в репозитории на GitHub (или через **GitHub Projects**)
2. Найдите задачу, которую хотите взять (по меткам `feature`, `bug` и т. д.)
3. Убедитесь, что задача в приоритете

#### Шаг 2. Назначьте себя на задачу

1. Откройте выбранную задачу (Issue)
2. В правой колонке нажмите кнопку **Assign yourself**
3. Теперь вы — ответственный за эту задачу. Другие разработчики увидят, что она занята

#### Шаг 3. Добавьте задачу в рабочий процесс (GitHub Projects)

1. Перейдите на доску **GitHub Projects** вашего проекта
2. Найдите колонку, соответствующую текущему этапу (например, «To Do» или «In Progress»)
3. Перетащите карточку задачи (Issue) в нужную колонку
4. При необходимости добавьте метки или укажите сроки выполнения

#### Шаг 4. Создайте ветку для работы (если её ещё нет)

**Если задача новая и для неё ещё нет ветки:**

1. Обновите локальную ветку `development`:
```bash
git checkout development
git pull origin development
```
2. Создайте новую ветку по шаблону:
    * для фичи: `feature/<краткое-описание-фичи>` (например, `feature/login-system`);
    * для бага: `fix/<краткое-описание-бага>` (например, `fix/auth-timeout`).
3. Выполните команды:
```bash
git checkout -b feature/login-system  # или fix/auth-timeout
git push origin feature/login-system  # создаём ветку на удалённом репозитории
```

**Если ветка для задачи уже создана кем‑то другим:**

1. Найдите название ветки в описании задачи (Issue) или на доске GitHub Projects.
2. Подключитесь к существующей ветке:
```bash
git fetch origin
git checkout feature/login-system  # укажите имя существующей ветки
```
3. Синхронизируйте ветку с удалённым репозиторием:
```bash
git pull origin feature/login-system
```

#### Шаг 5. Работайте в ветке

1. Вносите изменения в код (фронтенд в `frontend/`, бэкенд в `backend/` и т. д.).
2. Делайте коммиты с понятными сообщениями на английском языке. Используйте префиксы:
    * `feature(frontend): ...` — новая функциональность фронтенда;
    * `feature(backend): ...` — новая функциональность бэкенда;
    * `fix(frontend): ...` / `fix(backend): ...` — исправления;
    * `docs: ...` — изменения в документации;
    * `style: ...` — форматирование кода.

    **Пример:**
```bash
git commit -m "feat(frontend): Add login form UI"
git commit -m "feat(backend): Implement JWT authentication"
```
3. Регулярно запушивайте изменения в удалённую ветку:
```bash
git push origin feature/login-system
```

#### Шаг 6. Перед созданием Pull Request (PR)

Проверьте, что код работает локально:
  * запустите бэкенд (`rails s` в `backend/`)
  * запустите фронтенд (`npm start` в `frontend/`)
  * протестируйте функциональность

#### Шаг 7. Создайте Pull Request (PR)

1. Перейдите на страницу репозитория GitHub
2. Нажмите кнопку **Compare & pull request**
3. Заполните шаблон PR:
    * заголовок: кратко опишите суть изменений (например, «Add user login system»);
    * описание: укажите, какую задачу решает PR (ссылку на Issue), кратко опишите изменения и возможные нюансы;
4. Укажите рецензентов (reviewers) - тимлид и тестировщик
5. Выберите ветку назначения: **development**
6. Нажмите **Create pull request**

#### Шаг 8. Ревью и мердж

1. Дождитесь комментариев и замечаний от рецензентов (тимлид и тестировщик)
2. Внесите необходимые правки в ту же ветку (`feature/...` или `fix/...`)
3. После одобрения PR будет слит в ветку `development`
4. GitHub автоматически закроет связанный Issue после мерджа

---

**Важные замечания:**

* **Коммуникация.** Если у вас возникли вопросы по задаче, пишите в комментариях к Issue или в командном чате (ВК)
* **Одна задача - одна ветка**. Не смешивайте несколько фич/фиксов в одной ветке
* **Синхронизация.** Если в процессе работы кто‑то ещё внёс изменения в `development`, и ваша ветка устарела, синхронизируйте её:
```bash
git checkout feature/login-system
git fetch origin
git rebase origin/development
# Разрешите конфликты, если они возникли
git push --force-with-lease origin feature/login-system
```
* **Чистота веток.** После мерджа PR ветка `feature/...` или `fix/...` будет удалена автоматически (или вручную тимлидом). Не работайте в старых ветках

Если что‑то непонятно в процессе — обращайтесь к тимлиду

### Настройка и запуск проекта

#### Шаг 1. Клонируем проект

```bash
git clone https://github.com/GHKaktus/AutoShop/issues
cd <путь-к-проекту>
```

#### Шаг 2. Настраиваем и запускаем бэкенд (Ruby on Rails)

```bash
cd backend
cp .env-example .env
# Отредактируйте .env и заполните нужные значения
bundle install               # Установка зависимостей Ruby
rails db:create              # Создание базы данных
rails db:migrate             # Применение миграций
rails s                      # Запуск сервера
```
Бэкенд запустится на `http://localhost:3000`.

#### Шаг 3. Настраиваем и запускаем фронтенд (React)

Откройте новое окно терминала и выполните:
```bash
cd frontend
npm install                    # Установка зависимостей React
npm start                     # Запуск сервера разработки
```
Фронтенд будет доступен на `http://localhost:3001`.

#### Шаг 4. Запускаем Python‑mock‑сервер (опционально)

Используйте, если нужно протестировать API без готовой ответчной части (фронтенд или бекенд):
```bash
cd tests/mock-server
pip install -r requirements.txt  # Установка зависимостей Python
./run.sh                         # Запуск mock‑сервера
```
Mock‑сервер будет работать на `http://localhost:3002`.

Чтобы фронтенд использовал mock‑сервер, создайте файл `frontend/.env.local` со строкой:
```env
VITE_API_URL=http://localhost:3002
```

### Полезные команды — шпаргалка

| Действие | Команда |
|--------|--------|
| Запуск бэкенда | `cd backend && rails s` |
| Запуск фронтенда | `cd frontend && npm start` |
| Запуск mock‑сервера | `cd tests/mock-server && ./run.sh` |
| Миграции БД | `cd backend && rails db:migrate` |
| Установка зависимостей бэкенда | `cd backend && bundle install` |
| Установка зависимостей фронтенда | `cd frontend && npm install` |

### Если что‑то пошло не так

* **«Порт занят»** — проверьте, не запущены ли уже серверы на этих портах. Завершите процессы или используйте другие порты.
* **«Зависимости не установлены»** — убедитесь, что выполнили `bundle install` и `npm install` в нужных папках.
* **«База данных не найдена»** — проверьте `DATABASE_URL` в `backend/.env` и убедитесь, что PostgreSQL запущен.
* **«Mock‑сервер не отвечает»** — проверьте, что запустили `./run.sh` и что сервер слушает порт 3002.
